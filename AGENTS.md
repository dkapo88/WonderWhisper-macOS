# Repository Guidelines

WonderWhisper Mac is a SwiftUI macOS client. Use this guide to ramp quickly and keep contributions consistent.

## Project Structure & Module Organization
- `WonderWhisper Mac/`: app sources, SwiftUI views, `Resources/Assets.xcassets`, `Info.plist`, entitlements.
- `WonderWhisper MacTests/`: unit tests targeting pure logic, helpers under the same namespace.
- `WonderWhisper MacUITests/`: UI and flow tests using XCTest UI APIs.
- `WonderWhisper Mac.xcodeproj`: main project; derived data writes to `DerivedData_WW/` during local builds.
- `build/`: local artifacts generated by `xcodebuild`; safe to clean if space is needed.

## Build, Test, and Development Commands
- `open "WonderWhisper Mac.xcodeproj"` launches the project in Xcode.
- `xcodebuild -project "WonderWhisper Mac.xcodeproj" -scheme "WonderWhisper Mac" -configuration Debug build` creates a debug build from the command line.
- `xcodebuild -project "WonderWhisper Mac.xcodeproj" -scheme "WonderWhisper Mac" -destination 'platform=macOS' test` runs the XCTest suites.
- `open build/Debug/WonderWhisper\ Mac.app` starts the latest debug build outside Xcode; adjust the path if you changed the configuration.

## Coding Style & Naming Conventions
- Swift uses 2-space indentation and ~100 character lines; wrap earlier for clarity.
- Types adopt `PascalCase`; functions, variables, and bindings use `camelCase`; constants prefer `static let`.
- Keep one primary type per file and align filenames (`AudioTranscriber.swift`, `SettingsView.swift`).
- Favor small SwiftUI views with previews; avoid force unwraps—prefer `guard` statements.
- Run `swiftformat .` and `swiftlint` before publishing changes when tooling is available.

## Testing Guidelines
- XCTest underpins both unit and UI tests; prefer async/await-friendly APIs where possible.
- Name methods `test_<UnitUnderTest>_<Behavior>()`, e.g. `test_AudioService_handlesPermissionDenied`.
- Target core audio/transcription logic first, then UI flows; aim for ≥80% coverage on central modules.
- Execute `xcodebuild ... test` or Xcode’s Test action before submitting a PR.

## Commit & Pull Request Guidelines
- Craft commits that are small, focused, and imperative: `fix: handle microphone permission denial`.
- PRs should link related issues, outline approach, call out risks, and include screenshots or GIFs for UI changes.
- Confirm build, tests, and lint all succeed locally; note any gaps explicitly in the PR description.

## Security & Configuration Tips
- Never commit secrets; store environment values in local `.xcconfig` files or the Keychain.
- Audit entitlements and Hardened Runtime settings to minimize privileges.
- Avoid private macOS APIs and review third-party dependencies periodically.
