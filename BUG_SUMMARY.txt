================================================================================
CRITICAL BUG: Parakeet Local Whisper + Audio Enhancement = Zero/Empty Output
================================================================================

SYMPTOM:
--------
When Audio Enhancement is ON + Parakeet (local model) selected:
  Result: Empty transcription or 1-2 words only
  
When Audio Enhancement is ON + Groq/OpenAI selected:
  Result: Works perfectly

QUICK DIAGNOSIS:
----------------
ParakeetTranscriptionProvider.swift uses BROKEN file-based preprocessing:
  1. Reads audio file
  2. Decodes → Processes → Re-encodes to _proc.wav (decode #1)
  3. Reads _proc.wav from disk
  4. Decodes _proc.wav again (decode #2) ← DOUBLE DECODE
  5. Runs VAD on OVER-NORMALIZED audio
  6. VAD trims too aggressively
  7. ASR gets insufficient audio → EMPTY OUTPUT

Meanwhile, GroqTranscriptionProvider.swift uses CORRECT in-memory preprocessing:
  1. Reads audio file
  2. Decodes → Processes → RETURNS DATA (in memory)
  3. Sends data directly to API (no re-decoding)
  4. API handles transcription properly ✓

ROOT CAUSE:
-----------
File: ParakeetTranscriptionProvider.swift
Lines: 138-175, specifically 143-159 and 209-216

The bug has MULTIPLE failures:

1. LINE 152: Uses AudioPreprocessor.processIfEnabled() 
   ✗ Wrong: File-based processing (WAV disk I/O)
   ✓ Correct: Should use processToData() (in-memory)

2. LINE 209-216: Skips Parakeet's internal normalization
   ✗ Wrong: Assumes preprocessed audio is already correct
   ✓ Correct: Should re-normalize to Parakeet's expected levels

3. LINE 214: Expects audio at 0.06 RMS
   ✗ Wrong: AudioPreprocessor uses 0.08 RMS
   ✓ Correct: Should normalize after decode

4. LINE 221-223: VAD runs on wrong amplitude range
   ✗ Wrong: VAD thresholds don't match audio level
   ✓ Correct: VAD should be skipped after preprocessing

WHY ONLY PARAKEET BREAKS:
------------------------
Parakeet is EXTREMELY sensitive to audio levels:
  - VAD has strict thresholds
  - Auto-detect depends on SNR estimation  
  - Expects specific RMS levels (0.06)
  
Groq/OpenAI are robust because:
  - They handle their own preprocessing (cloud)
  - They accept raw audio data
  - Flexible to level variations

KEY FILES:
----------
ParakeetTranscriptionProvider.swift    (BROKEN)
  Lines 138-273: transcribe() function
  Lines 143-159: Bad preprocessing integration
  Lines 209-216: Double-normalization bug
  Lines 221-223: Aggressive VAD failure
  
AudioPreprocessor.swift                (SUPPORTING)
  Lines 26-72:   processIfEnabled() (file-based)
  Lines 196-258: processToData() (in-memory, NOT USED by Parakeet)
  Lines 182-183: Normalization values (0.08 RMS)

GroqTranscriptionProvider.swift        (REFERENCE - CORRECT)
  Lines 31-38:   Uses processToData() ← THE RIGHT WAY
  
OpenAITranscriptionProvider.swift      (REFERENCE - CORRECT)
  Lines 29-36:   Uses processToData() ← THE RIGHT WAY

QUICK FIX:
----------
In ParakeetTranscriptionProvider.swift, replace lines 143-159 with code that:
  1. Calls AudioPreprocessor.processToData() instead of processIfEnabled()
  2. Writes result to temp file (keep same interface)
  3. Sets preprocessingApplied=true appropriately

This ensures audio is decoded only ONCE from preprocessing, not twice.

COMPLETE FIX ALSO REQUIRES:
  1. Skip VAD when preprocessingApplied=true (different amplitude levels)
  2. Or re-normalize audio after decoding to Parakeet's expected levels
  3. Better error handling for empty audio
